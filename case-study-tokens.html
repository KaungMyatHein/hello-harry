Act 2 · Research

The Discovery

The gap isn't between Figma and code. The gap is between Figma variables and platform-specific naming conventions.

Designers can define tokens in Figma. But exporting them reliably to CSS, JavaScript, Swift, Kotlin, Flutter, and JSON—each with its own naming rules—is the actual problem. Bridge that, and the handoff works.

Act 3 · Design

Smart Controls

Collections: Select which Figma variable groups to export. Filter by color, spacing, typography, or shadows.

Modes: Include light/dark/custom theme variations in a single export. Choose which modes to include.

Naming: camelCase for JavaScript, PascalCase for Swift, snake_case for Python/SQL. Applied on export, not in Figma.

Format: CSS/SCSS variables, JavaScript objects, Dart constants, Kotlin objects, Swift structs, or raw JSON.

Act 4 · Results

5 Technical Lessons

Lesson 1
Preserve Original Casing
Store token names as-is in Figma. Apply naming conventions only at export. Never overwrite the source.
Lesson 2
Test with Multi-Segment Names
Names like "color-primary-hover-dark" expose casing logic bugs. Test edge cases early.
Lesson 3
Preview Catches Errors
Real-time preview validation eliminates most export bugs before they ship.
Lesson 4
Proxy Simplifies CORS
Local server handles Notion API auth. Eliminates cross-origin headaches.
Lesson 5
Smart Defaults Win Adoption
Default to platform conventions. camelCase for JS, snake_case for Python. Less thinking = more adoption.
