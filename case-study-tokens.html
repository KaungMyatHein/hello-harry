Act 2 · Research

The Discovery

4
Research Phases
Designer interviews → Developer sessions → Pain point synthesis → Feature validation. Interviewed 4 designers and 6 developers. Key finding: naming mismatches were the #1 design-to-code handoff frustration. Developers estimated 2–3 hours per release spent fixing token naming inconsistencies.
💡
The Insight
The gap isn't between Figma and code. The gap is between Figma variables and platform-specific naming conventions.

Designers can define tokens in Figma. But exporting reliably to CSS, JS, Swift, Kotlin, Flutter, and JSON — each with its own naming rules — is the actual problem.

Act 4 · Results

5 Technical Lessons

Lesson 01
Preserve Original Casing
Store token names as-is in Figma. Apply naming conventions only at export. Never overwrite the source.
Lesson 02
Test Multi-Segment Names
"color-primary-hover-dark" exposes casing logic bugs. Test edge cases early and often.
Lesson 03
Preview Catches Errors
Real-time preview validation eliminates most export bugs before they ship. Zero-cost error detection.
Lesson 04
Proxy Simplifies CORS
Local server handles Notion API auth. Eliminates cross-origin headaches for plugin-to-API communication.
Lesson 05
Smart Defaults Win Adoption
Default camelCase for JS, snake_case for Python. Less thinking = more adoption. Conventions reduce friction.
